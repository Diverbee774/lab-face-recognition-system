# 软件需求规格说明书 (SRS)

**项目名称**：基于人脸识别的实验室智能出入管理系统
**版本号**：V1.0
**编制日期**：2025-12-05

-----

## 1\. 引言

### 1.1 编写目的

本文档详细描述了“基于人脸识别的实验室智能出入管理系统”的功能需求、非功能性需求及接口规范。该文档旨在为开发团队提供明确的实施依据，明确系统边界，并作为项目验收的核心标准。

### 1.2 项目背景

针对高校及科研机构多实验室场景下人员流动性大、管理困难的问题，本项目旨在构建一套基于计算机视觉与微服务架构的智能门禁系统。系统通过人脸识别技术替代传统钥匙或IC卡，实现无感通行，并结合后台权限管理机制，实现对不同实验室的精细化管控（如公共开放与私有授权）。

### 1.3 适用范围

本系统适用于需要对多个独立物理空间进行差异化权限控制的场景，如高校教学实验室、科研课题组工作室、机房等。

-----

## 2\. 系统总体描述

### 2.1 技术架构

系统采用 B/S 架构与前后端分离模式，后端采用异构服务协同：

  * **表现层 (Web)**：Vue.js + Element UI，负责管理后台与门禁模拟终端展示。
  * **业务层 (Java)**：Spring Boot，负责业务逻辑、权限控制、数据库交互及文件存储。
  * **算法层 (Python)**：FastAPI + Face\_recognition，作为独立微服务，专注于人脸特征值的提取与比对。
  * **数据层**：MySQL（关系型数据）+ 本地文件系统（非结构化图像数据）。

### 2.2 用户角色

1.  **普通用户**：进行注册、人脸录入、查看个人通行记录。
2.  **系统管理员**：拥有最高权限，负责用户管理、实验室创建、门禁策略配置（白名单管理）、日志审计及数据看板监控。

## 3\. 功能需求详细说明

### 3.1 用户管理模块

本模块分为用户自服务与管理员管理两部分。

#### 3.1.1 注册与人脸采集 (用户端)

  * **基本信息注册**：用户使用学号/工号作为唯一ID进行注册。
  * **人脸底库录入**：用户上传标准免冠照片，系统调用 Python 服务进行质量检测（是否包含人脸），通过后保存至底库目录。

#### 3.1.2 用户全生命周期管理 (管理员端)

  * **用户列表查询**：管理员可分页浏览所有注册用户，支持按学号、姓名进行模糊搜索。
  * **账号状态控制**：
      * **禁用/启用**：管理员可将违规用户标记为“禁用”。被禁用的用户即使人脸匹配成功，也会被业务层拦截，禁止通行。
      * **重置密码**：管理员可重置遗忘密码的用户账户。
  * **底库管理与删除**：
      * **强制删除**：管理员可删除离校用户。删除时系统需同步执行：删除数据库记录、物理删除底库照片、调用接口通知 Python 服务刷新内存特征库。
      * **照片重置**：对于识别率低的用户，管理员可清除其底库照片，强制其下次登录时重新上传。

### 3.2 实验室与策略管理模块 (管理员端)

本模块决定了“谁能进哪里”，是系统的核心权限配置中心。

#### 3.2.1 实验室基础维护

  * **实验室CRUD**：管理员可创建、编辑、删除实验室信息。
  * **属性定义**：
      * **实验室ID/名称**：唯一标识（如“305大数据实验室”）。
      * **最大容量**：用于限流预警。
      * **运营状态**：可切换为“开放中”或“维护中”（维护状态下除管理员外禁止进入）。

#### 3.2.2 访问策略配置

管理员需为每个实验室配置访问模式：

1.  **公共模式 (Public)**：
      * **逻辑**：所有状态正常的注册用户均可通行。
      * **场景**：公共机房、图书馆、自习室。
2.  **私有模式 (Private)**：
      * **逻辑**：仅限被添加至该实验室“白名单”内的用户通行。
      * **场景**：导师课题组、存放贵重设备的实验室。

### 3.3 白名单权限管理模块 (管理员端)

针对配置为“私有模式”的实验室，管理员需进行人员授权管理。

  * **R-3.3.1 实验室人员概览**：选择特定实验室，列表展示当前拥有该实验室进入权限的所有用户。
  * **R-3.3.2 批量授权 (添加白名单)**：
      * 支持从全校用户池中勾选多人，批量添加至当前实验室。
      * 支持通过学号精确查找并添加。
  * **R-3.3.3 权限移除**：
      * 管理员可将用户从当前实验室白名单中移除（撤销权限）。

### 3.4 智能通行控制模块 (核心业务)

#### 3.4.1 识别流程

1.  **终端抓拍**：门禁终端（前端页面）捕获视频流帧，上传至 Java 后端。
2.  **人脸比对**：Java 后端将图片路径发送至 Python 服务，Python 返回识别到的 UserID。
3.  **多级鉴权 (Java后端逻辑)**：
      * **Level 1 (身份存在)**：Python 返回非“Unknown”，且数据库中存在该 UserID。
      * **Level 2 (账号合规)**：检查该 UserID 的账号状态是否为“启用”。
      * **Level 3 (实验室状态)**：检查目标实验室是否处于“维护中”。
      * **Level 4 (权限匹配)**：
          * 若实验室为 **公共模式** -\> 允许通行。
          * 若实验室为 **私有模式** -\> 查询 `lab_user_relation` 表，检查该 UserID 是否在当前实验室白名单内 -\> 存在则允许，否则拒绝。

#### 3.4.2 结果反馈

系统根据鉴权结果返回状态码与提示信息（如：“欢迎进入”、“未授权人员”、“账号已封禁”），并在前端展示对应颜色的提示框。

### 3.5 审计与监控模块

  * **R-3.5.1 全局出入日志**：记录所有尝试通行的事件。字段包含：时间、地点、人员ID（若识别成功）、抓拍照片快照、通行结果（成功/失败）、失败原因。
  * **R-3.5.2 数据可视化大屏**：
      * **实时流量**：今日累计通行人次。
      * **安全告警**：今日拦截陌生人/无权限访问次数。
      * **热力图**：各实验室当前占用率排行。

## 4\. 接口与数据交互规范

### 4.1 内部协同接口 (Java -\> Python)

**原则**：Python 仅负责生物特征识别（Who are you），Java 负责业务权限判断（Can you enter）。

1.  **人脸识别接口**

      * **URL**: `/api/face/recognize`
      * **Method**: `POST`
      * **Request**:
        ```json
        { "image_path": "/data/snapshots/20251205_cam01_001.jpg" }
        ```
      * **Response**:
        ```json
        {
          "code": 200,
          "data": {
            "user_id": "202100301",
            "confidence": 0.45
          },
          "msg": "success"
        }
        ```

2.  **特征库重载接口**

      * **URL**: `/api/face/reload`
      * **Method**: `POST`
      * **描述**：当 Java 端完成用户底库照片的增删改后，调用此接口通知 Python 重新加载特征向量。

### 4.2 目录结构规范

  * `/app/data/faces/base/`：存放注册底库照片（命名格式：`studentID.jpg`）。
  * `/app/data/faces/daily/`：存放每日通行抓拍照片（按日期分文件夹存储，定期归档清理）。

## 5\. 非功能性需求

### 5.1 性能要求

  * **识别速度**：从图片上传到返回鉴权结果，全链路延迟不超过 **1秒**。
  * **并发能力**：支持至少 **20个** 门禁终端同时发起请求不阻塞。

### 5.2 安全性

  * **接口鉴权**：所有管理端 API 需通过 JWT (JSON Web Token) 校验管理员身份。
  * **隐私保护**：抓拍照片与底库照片不直接通过静态资源映射对外暴露，需通过后端流式接口读取，防止遍历爬取。

### 5.3 可靠性

  * **异常处理**：当 Python 服务不可用时，Java 端应捕获异常并返回“系统维护中”提示，避免前端页面卡死。
